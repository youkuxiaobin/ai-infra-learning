# 03 内存模型与规约优化

## 1. 为什么逐元素加法可以并行

每个线程只操作 `A[i]`、`B[i]`、`C[i]`，没有共享数据，这种并行属于 **Embarrassingly Parallel（天然并行）**。

```cpp
C[i] = A[i] + B[i];
```

## 2. 并行累加的问题：数据竞争

并行求和和逐元素加法不同，会有多个线程同时写同一个结果地址，因此会产生 **数据竞争**。

最直接的做法是使用 `atomicAdd()`，保证一个线程写完后，下一个线程再写。

### 不安全版本

```cpp
template<typename T>
__global__ void sum_kernel(T* result, const T* input, size_t n)
{
    size_t idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n)
        *result += input[idx];
}
```

### 原子加版本

```cpp
template<class T>
__global__ void atomic_sum_kernel(T* result, const T* input, size_t n){
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        atomicAdd(result, input[idx]);
    }
}
```

## 3. 优化思路：减少 atomic 次数

课件的核心思路是：

- 原始方案：`n` 次 `atomicAdd`
- 优化方案：先在 warp 或 block 内做局部求和，再把局部结果做原子加
- 例如一个 warp 有 32 个线程，先做 warp 内求和，再由 warp leader 执行一次 `atomicAdd`

也就是把原来的 `n` 次 atomic，降低到大约 `n / 32` 次，甚至更少。

## 4. Warp Reduction

### warp 级求和思路

只有 `lane_id == 0` 的线程负责收集一个 warp 内 32 个线程的和，然后执行一次 `atomicAdd`。

```cpp
template<class T>
__global__ void reduce_warp_sum(T* result, const T* input, size_t n){
    int tid = threadIdx.x;
    int idx = blockDim.x * blockIdx.x + threadIdx.x;

    int lane_id = tid % 32;
    if (lane_id == 0) {
        int warp_sum = 0;
        for(int i = 0; i < 32; i++) {
            for(int m = idx + i; m < n; m += gridDim.x * blockDim.x) {
                warp_sum += input[m];
            }
        }
        atomicAdd(result, warp_sum);
    }
}
```

## 5. Block 级局部规约版本

和 warp 版类似，但粒度提升到 block：

- 每个 block 做局部求和
- block 之间再通过 `atomicAdd` 汇总

```cpp
template <typename T>
__global__ void reduce_atomic_kernel(T* output, const T* input, size_t n) {
    size_t tid = threadIdx.x;
    size_t idx = blockIdx.x * blockDim.x + tid;

    T block_sum = 0;
    for (size_t i = idx; i < n; i += blockDim.x * gridDim.x) {
        block_sum += input[i];
    }

    atomicAdd(output, block_sum);
}
```

## 6. Shared Memory Reduction

进一步优化的思路是：

1. 每个线程先在寄存器里累加自己的局部和
2. 把局部和写入 shared memory
3. 在 block 内做树状规约
4. 每个 block 最终只执行一次 `atomicAdd`

### block 内 shared memory 规约

```cpp
template <typename T>
__global__ void reduce_block_kernel(T* output, const T* input, size_t n) {
    extern __shared__ T sdata[];

    unsigned int tid = threadIdx.x;
    unsigned int idx = blockIdx.x * blockDim.x + tid;
    unsigned int stride = blockDim.x * gridDim.x;

    T thread_sum = 0;
    for (size_t i = idx; i < n; i += stride) {
        thread_sum += input[i];
    }

    sdata[tid] = thread_sum;
    __syncthreads();

    // block 内规约
    for (unsigned int s = blockDim.x / 2; s > 0; s >>= 1) {
        if (tid < s) {
            sdata[tid] += sdata[tid + s];
        }
        __syncthreads();
    }

    // 每个 block 只做一次 atomicAdd
    if (tid == 0) {
        atomicAdd(output, sdata[0]);
    }
}
```

### 课件中的存储层级说明

- `register memory`：每个 thread 私有
- `shared memory`：每个 block 内共享
- `L1 cache memory`：每个 SM 内部共享，可被多个 block 使用
- `L2 cache memory`：整个 GPU 共享

## 7. 树状规约示意

课件给出的树状规约流程如下：

```text
t0 t1 t2 t3 t4 t5 t6 t7

step1
 t0 += t4
 t1 += t5
 t2 += t6
 t3 += t7

step2
 t0 += t2
 t1 += t3

step3
 t0 += t1

最终：
 t0 = sum
```

对应 kernel 开头如下：

```cpp
template<typename T>
__global__ void reduce_smem_kernel(T* output,
                                   const T* input,
                                   size_t n)
{
    extern __shared__ T smem[];

    size_t tid = threadIdx.x;
    size_t idx = blockIdx.x * blockDim.x + tid;

    smem[tid] = (idx < n) ? input[idx] : 0;
```

## 8. kernel launch 与 Warp Shuffle

### 启动方式

```cpp
kernel<<<grid, block, shared_size>>>()
```

### Warp Shuffle

CUDA 提供 Warp Shuffle 指令，可以让 warp 内线程**直接交换寄存器数据**。

```cpp
__shfl_down_sync()
```

课件强调：warp 内 32 个线程是 **lockstep** 执行的，也就是同一时刻执行同一条指令。因此：

```cpp
val += __shfl_down_sync(...)
```

本质上是一条 warp 指令完成的线程间数据交换与累加。

### 例子

- warp0 的 32 个线程同时执行 shuffle 指令
- `lane0` 读取 `lane16`
- `lane1` 读取 `lane17`
- `lane2` 读取 `lane18`
- ...

## 9. warp_reduce 函数

```cpp
template<typename T>
__inline__ __device__
T warp_reduce(T val)
{
    for (int offset = 16; offset > 0; offset /= 2)
        int temp = __shfl_down_sync(0xffffffff, val, offset);
    if (threadIdx.x < offset) {
        val += temp;
    }

    return val;
}
```

课件意图是：通过不断让前半部分线程读取后半部分线程的数据，实现一个 warp 内的规约。

## 10. 基于 Warp Shuffle 的规约 kernel

### warp 级版本

```cpp
template<typename T>
__global__ void reduce_shuffle_kernel(T* output,
                                      const T* input,
                                      size_t n)
{
    size_t tid = threadIdx.x;
    size_t idx = blockIdx.x * blockDim.x + tid;

    T sum = 0;

    for (size_t i = idx; i < n; i += blockDim.x * gridDim.x)
        sum += input[i];

    sum = warp_reduce(sum);

    if ((tid & 31) == 0)
        atomicAdd(output, sum);
}
```

## 11. 抽象出 `warp_reduce()` 的 block 级优化版本

课件总结出的优化路径：

- 抽象一个 `warp_reduce()`
- 使用 `#pragma unroll`
- 先做 warp 内规约
- 再用 shared memory 做 warp 间规约
- 最后每个 block 只做一次原子加

### `warp_reduce()`

```cpp
template <typename T>
__device__ T warp_reduce(T val) {
#pragma unroll
    for (int offset = 16; offset > 0; offset >>= 1) {
        val += __shfl_down_sync(0xffffffff, val, offset);
    }
    return val;
}
```

### block 级版本

```cpp
template <typename T>
__global__ void reduce_warp_shuffle_kernel(T* output, const T* input, size_t n) {
    __shared__ T smem[32];

    size_t tid = threadIdx.x;
    size_t idx = blockIdx.x * blockDim.x + tid;

    T sum = 0;
    for (size_t i = idx; i < n; i += blockDim.x * gridDim.x) {
        sum += input[i];
    }

    sum = warp_reduce(sum);

    if (tid % 32 == 0) {
        smem[tid / 32] = sum;
    }
    __syncthreads();

    if (tid < 32) {
        T block_sum = (tid < (blockDim.x + 31) / 32) ? smem[tid] : 0;
        block_sum = warp_reduce(block_sum);

        if (tid == 0) {
            atomicAdd(output, block_sum);
        }
    }
}
```

## 12. 逐行解释：`warp_reduce()` 的意义

### `__device__ T warp_reduce(T val)`

这是一个设备函数，只能在 GPU 端被其他 kernel 调用。

它的作用是：

- 把一个 warp 内 32 个线程的 `val` 规约成 `lane 0` 的总和

### `#pragma unroll`

告诉编译器把这个小循环展开。课件给出的理解方式是把：

```cpp
for (int offset = 16; offset > 0; offset >>= 1)
```

展开成近似下面这样的形式：

```cpp
val += __shfl_down_sync(..., 16);
val += __shfl_down_sync(..., 8);
val += __shfl_down_sync(..., 4);
val += __shfl_down_sync(..., 2);
val += __shfl_down_sync(..., 1);
```

这样做的目的，是减少循环控制开销。

## 13. 为什么 `__shared__ T smem[32]` 足够

课件给出的解释是：

- 一个 block 最多 1024 个线程
- 一个 warp 有 32 个线程
- 所以最多有 `1024 / 32 = 32` 个 warp
- 因此 `smem[32]` 足够存放每个 warp 的部分和

例如：

- 如果一个 block 有 256 线程，那么共有 8 个 warp
- 每个 warp 的 `lane 0` 把结果写入 `smem[0..7]`
- `__syncthreads()` 之后，由 block 的第一个 warp 继续把这些部分和规约成一个 `block_sum`
- 最后只由 `tid == 0` 的线程执行一次 `atomicAdd(output, block_sum)`

## 14. Shared Memory 不一定永远快

课件强调：

- shared memory 很快，但不是无条件快
- 只有访问模式好的时候才快
- 如果访问模式导致 **bank conflict**，shared memory 也会明显变慢

## 15. Bank Conflict

课件把 shared memory 比喻成一个银行大厅：

- 每个窗口 = 一个 bank
- 一个 warp 的 32 个线程 = 32 个客户
- 如果 32 个线程分别访问不同 bank，就能并行完成
- 如果很多线程同时访问同一个 bank 的不同地址，就会排队，这就是 **bank conflict**

### bank 映射的直觉模型

对于 4 字节数据类型（例如 `float`），可以近似理解为：

```cpp
bank_id = (address / 4) % 32
```

如果访问的是 `float smem[i]`，那么可以进一步近似看成：

```cpp
bank_id = i % 32
```

例如：

```text
smem[0]  -> bank 0
smem[1]  -> bank 1
...
smem[31] -> bank 31
smem[32] -> bank 0
smem[33] -> bank 1
```

## 16. 什么情况不会冲突

这一类访问通常很友好：

```cpp
smem[threadIdx.x]
```

即：

```text
线程 0 访问 smem[0]
线程 1 访问 smem[1]
...
线程 31 访问 smem[31]
```

因为每个线程访问不同的 bank，所以：

- 没有 bank conflict

课件也提到，下面这种写法通常是很友好的：

```cpp
smem[tid] = input[idx];
```

## 17. 什么情况会发生 bank conflict

一个典型例子是：

```cpp
smem[threadIdx.x * 2]
```

此时：

```text
线程 0  -> smem[0]  -> bank 0
线程 1  -> smem[2]  -> bank 2
线程 2  -> smem[4]  -> bank 4
...
线程 16 -> smem[32] -> bank 0
线程 17 -> smem[34] -> bank 2
```

这样会出现：

- 线程 0 和线程 16 都访问 bank 0
- 线程 1 和线程 17 都访问 bank 2
- ...

这就会形成 **2-way bank conflict**。

课件的解释是：

- 原本一次 warp 访问可以一拍完成
- 发生 bank conflict 后，会被拆成两拍
- 因而访问变慢

## 18. 本节核心总结

这份课件围绕 CUDA reduction 的优化路径，给出的主线是：

1. 先理解为什么逐元素操作容易并行，而求和会有写冲突
2. 用 `atomicAdd` 得到最直接但代价较高的正确实现
3. 通过 warp 级、block 级局部规约减少 atomic 次数
4. 利用 shared memory 做 block 内树状规约
5. 利用 warp shuffle 直接在寄存器间交换数据，减少 shared memory 和同步开销
6. 注意 shared memory 的访问模式，避免 bank conflict
